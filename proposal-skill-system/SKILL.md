# Proposal Deck from Analysis — SKILL

## このSkillの目的

プレセールス分析資料を入力として受け取り、3段階のアウトプット（①骨子 ②HTMLDeck ③PPT）を生成しながら、各段階で方針確認のウィジェットをかけてユーザーの判断を介して進める提案書ビルダー。19枚構成のテンプレートをデフォルトとし、F-2型（プラットフォーム型）以外のビジネスモデルでも適用できるよう汎用化されている。

---

## STEP 0: 入力受付と前提確認

### 受け取る入力

以下のいずれかの形で分析資料を受け取る：
- 5seg設計（5seg定義案・推測値・売上貢献率）
- ビジネスモデル分析（型判定・KSF・USP・3C情報）
- KPIツリー（KGI→KSF→KPI）
- CS分析・顧客体験分析（CS定義・各CSの行動・ページURL）
- ボトルネック分析（5seg転換間ボトルネックの優先度ランキング・課題コード）
- 施策検討資料（優先施策・対応課題・打ち手・期待効果・KPI）

### deck.md の取り扱い

ユーザーから `deck.md` というデザイン参照ファイルが渡された場合：
- `references/04-design-system.md` を読み込み、優先する
- deck.md にCSS変数・カラー・タイポグラフィ・レイアウトルールが定義されていればそれを使う
- 渡されていない場合は、`references/04-design-system.md` のフォールバックを使う

## STEP 1: 提案の骨子を生成 → ウィジェット確認

### 出力物

19枚分の骨子テキスト（Markdown形式）。各スライドについて：
- **論点**：そのスライドで答えるべき問い
- **タイトル**：20文字以内
- **メインメッセージ**：120字以内（厳守）。強調`<span>`は本当に効かせたい1点だけ
- **チャートイメージ**：図解の概念（フロー／対比／ピラミッド／表 等）と主要要素

### 19枚デフォルト構成

| # | タイトル候補 | 論点 | レイアウト |
|---|---|---|---|
| 1 | 表紙 | — | Cover |
| 2 | アジェンダ | 何を順に議論するか | Agenda |
| 3 | エグゼクティブサマリ | 結論一枚 | 課題/原因/打ち手 3行 |
| 4 | ビジネスモデル × 3C分析 | 構造と力点 | BM構造図 + 3Cカード |
| 5 | KSFと典型ボトルネック | 一般論×自社実態 | 5segカード + KSF対比 |
| 6 | KPIツリー（KGI→KSF→KPI） | 主戦場の特定 | 3階層KPIツリー |
| 7 | CS全景 × 5seg接続 | 壁が起きるステージ | 10ステージカード横並び |
| 8 | 顧客体験エラー一覧 | 離脱原因の分解 | 6列表（5seg/CS/カテゴリ/コード/3層エラー/原因） |
| 9 | 施策方針と4施策一覧 | 何をどこに打つか | 6列表（誰の/どのCS/どの体験/どの課題/打ち手/施策） |
| 10〜13 | 施策1〜4詳細 | 各施策の具体 | 左:アイコン+構造図 / 右:詳細説明 |
| 14 | 接続マトリクス | 論理整合性 | 5seg×CS×顧客体験×4施策の交差表 |
| 15 | 算出式ロジック | 試算の根拠 | KPIツリー再掲（左）+ 改善幅早見表（右） |
| 16 | 各施策の期待改善幅 | 定量効果 | 4施策×悲観/通常/楽観の表 |
| 17 | ROIシミュレーション | 投資回収 | 3シナリオROI |
| 18 | ロードマップ × LTV拡大 | 推進計画と二次効果 | 12ヶ月ガント+LTV拡大図 |

詳細は `references/01-deck-structure.md` を参照。

### コンテンツルール（厳守）

`references/02-content-rules.md` を読み込み、以下を必ず適用する：
- **メインメッセージ120字以内**
- **強調`<span>`は本当に効かせたい1点だけ**
- **5seg推測値は◎/○/△で含み表現**（具体％を直接引用しない。業界基準値はそのまま残す）
- **CS連番**：1.認知 / 2.会員登録済み未利用 / 3.比較・検討 / 4.初回予約完了 / 5.継続利用 / 6.ロイヤル / 7.離反
- **顧客体験カテゴリ連番**：1.サイト訪問 / 2.会員登録 / 3.条件絞り込み検索 / 4.商品詳細閲覧 / 5.申し込み・購入 / 6.実物確認 / 7.アプリ起動 / 8.レビュー投稿 / 9.関連サービス予約 / 10.メール開封・再訪
- **KPIツリー一貫性**：複数ページに登場するKPIツリーは完全に同一構造（KGI→KSF×4→KPI×8の3階層）

### ウィジェット質問（STEP 1終了時）

`references/03-widget-flow.md` の `widget_after_outline` を使う。骨子全体を提示してから以下を聞く：

```
質問1: 骨子の方針は問題ないですか？
- 進める（HTMLDeck生成へ）
- 一部修正したい
- 大幅に作り直したい

質問2 (進めるの場合のみ): どのスライドを最重要アクセントにしますか？
- 施策3など（ユーザー指定）

質問3 (修正の場合のみ): 修正したいスライドはどれですか？
- P1〜P19の番号で指定
```

## STEP 2: HTMLDeckを生成 → ウィジェット確認

### 前提

承認された骨子をベースに、HTMLDeck（19枚分のスライドが縦に並ぶ単一HTMLファイル）を生成する。ファイル名は `proposal_v1_19pages.html` のように versioning する。

### デザインシステム

`references/04-design-system.md` のCSS変数・カラー・タイポグラフィ・レイアウトを使用する。`deck.md` が提供されていればそれが優先。

主要レイアウトテンプレート（10種）：Cover / Agenda / Executive Summary / KPI / 2カラム / 3カラム / Process Flow / Chart / Next Steps / Section Divider。詳細は consulting-slide-builder の `references/html-template-system.md` を参照（互換性あり）。

### SVG概念図ルール

- viewBox は基本 `0 0 800 380`、フル画面で見せたいスライドは `0 0 1000 500`
- 配色：グレー基調（`#F0F0F0`/`#E0E0E0`/`#999`）+ アクセント（`#02BDBD` または `#2563EB`）+ 注目色（`#E04E49`）
- 文字は日本語、最小11px、`<tspan>` で2行折り返しを許可（フォントを下げない）
- 装飾ゼロ（シャドウ・グラデーション・枠線過剰使用禁止）

### 自己チェック（生成後に必ず実行）

- [ ] 全スライドのメインメッセージが120字以内か
- [ ] 5seg推測値が直接引用されていないか（◎/○/△で含み表現になっているか）
- [ ] CS表記に連番が付いているか
- [ ] 顧客体験カテゴリに連番が付いているか
- [ ] KPIツリーが複数ページで同一構造か（特にP6とP15）
- [ ] 強調`<span>`が各ページで多くても1箇所か
- [ ] フッター・ページ番号が全スライドに入っているか

### ウィジェット質問（STEP 2終了時）

`references/03-widget-flow.md` の `widget_after_html` を使う：

```
質問1: HTMLDeckのデザイン・内容は問題ないですか？
- 進める（PPT生成へ）
- スライドの内容を一部修正したい
- デザインを修正したい
- デザインルール（deck.md）を読み込んで再生成したい

質問2 (修正の場合): 修正対象のスライドと修正内容を教えてください（自由記述）
```

## STEP 3: 編集可能PPTを生成

### 出力物

`proposal_v1_19pages.pptx` を `/mnt/user-data/outputs/` に保存。

### 絶対要件（ユーザー要件由来）

1. **編集可能なPPT**：すべての要素は PPTXネイティブシェイプ／テキストフレームとして配置する。背景画像（PNG焼き込み）は使わない。
2. **オブジェクトに直接文字を書く**：rect等のシェイプの上にテキストを重ねるのではなく、シェイプの `text_frame` にテキストを直接埋め込む。

### 変換手順

`references/05-pptx-conversion.md` の手順に従う。同梱スクリプトを順番に実行：

```bash
# STEP A: HTML → JSON 抽出（Playwrightで全要素の座標を取得）
python3 scripts/extract_layout.py

# STEP B: JSON → 編集可能PPTX 生成（python-pptxでネイティブシェイプ化）
python3 scripts/build_native_pptx.py
```

### 変換ルール（要点）

- HTML `<div>` → 矩形シェイプ + text_frame
- SVG `<rect>` → 矩形シェイプ
- **SVG `<text>` → 親rectのtext_frameに統合**（textの中心点でrect内判定 → 最も内側のrectに paragraph として追加）
- SVG `<line>` → 直線シェイプ（MSO_CONNECTOR.STRAIGHT）
- SVG `<polygon>` → 三角形シェイプ（MSO_SHAPE.RIGHT_TRIANGLE）
- SVG `<circle>` → 楕円シェイプ（MSO_SHAPE.OVAL）

### 自己チェック

- [ ] スライド枚数がHTMLDeckと同じ19枚か
- [ ] スライドサイズが16:9（13.33in × 7.5in）か
- [ ] 合計シェイプ数が500以上あるか（参考実績: 626）
- [ ] テキスト付きシェイプが全体の50%以上あるか（参考実績: 72%）
- [ ] PowerPointで開いてテキストが編集可能か（PNG背景になっていないか）

### 出力時の振る舞い

PPT生成完了後は **ウィジェット確認は不要**。`present_files` で PPT と HTMLDeck を並べて提示し、変更点があれば再度依頼してもらう旨を一言添える。

## エッジケース

| ケース | 対処 |
|---|---|
| ビジネスモデルがF-2型（プラットフォーム型）以外 | 「未顧客→トライアル」を該当する転換ボトルネックに置き換え。F-1型ならF0→F1、SaaSならトライアル→有料、D2CならF0→F1にマップ |
| 施策が4個未満 | 施策スライドを施策数だけにし、空白を Appendix にする旨を確認 |
| 施策が4個超 | 上位4施策のみ詳細スライド化し、残りは P9 の表に行追加 |
| KPIツリーのKSFが4本でない | KSF本数に合わせてレイアウト調整。3本なら横3カラム、5本なら2段組 |
| ROI試算ができない（数値根拠が薄い） | P15・P16・P17 を「対象人数 × 改善率 = 期待改善幅」の悲観/通常/楽観 3シナリオで表現し、推測値である旨を赤帯注記で明示 |
| ユーザーが「短いバージョン」を要求 | 19枚→10枚版（表紙・アジェンダ・エグゼ・KPI・CS全景・エラー一覧・4施策一覧・施策詳細1枚・ROI・ロードマップ）に圧縮 |

## PDCA反映ルール

案件実行後は `cases/` に入力、出力、レビュー、学びを残し、共通化できるものだけを `references/06-pdca-learnings.md` に昇格する。

### 5件実行後に追加された標準ルール

- P3は施策紹介ではなく、経営課題、主要ボトルネック、打ち手、事業インパクト、意思決定事項の順で構成する
- P6のKPIツリーはBM型に合わせて置き換える。D2C、SaaS、宿泊、人材紹介、店舗会員型は `references/06-pdca-learnings.md` の型を優先する
- P8のエラー一覧は網羅より優先順位を重視し、上位5件を中心に見せる
- P10-P13の施策詳細にはBefore/After、Owner、KPI、Effortを必ず含める
- P17のROIは売上増分だけでなく、粗利、手数料削減、広告費効率、運用工数を案件に応じて含める

## SKILL利用の典型フロー（ユーザー観点）

```
USER: 「Booking.comの分析資料をまとめて、提案書を作ってほしい」
      （5seg・KPIツリー・CS分析・施策検討の.mdファイルを添付）

CLAUDE: STEP 1: 19枚の骨子をMarkdownで生成
        → ウィジェット表示「骨子の方針は問題ないですか？」

USER: 「進める」を選択

CLAUDE: STEP 2: HTMLDeckを生成（120字以内/CS連番/◎○△/KPIツリー一貫性すべて適用）

USER: 「進める」を選択

CLAUDE: STEP 3: PPTに変換
        → present_files で HTMLDeck と PPTX を提示
        → 「修正があれば声をかけてください」と一言添えて完了
```
